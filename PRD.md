# PRD — Інтернет-магазин велосипедів та велозапчастин «VeloServer»

**Версія:** 1.2  
**Дата:** травень 2026  
**Статус:** Чернетка

---

## 1. Огляд продукту

### 1.1 Мета

Створити україномовний інтернет-магазин для продажу велосипедів та велозапчастин, орієнтований на українських покупців. Магазин має забезпечити зручний пошук товарів, оформлення замовлень, оплату та доставку по Україні.

### 1.2 Проблема

Більшість існуючих онлайн-магазинів велотоварів або не адаптовані під українського споживача (мова, оплата, доставка), або мають застарілий UX. Потрібне сучасне, швидке та зрозуміле рішення.

### 1.3 Цільова аудиторія

| Сегмент | Опис |
|---|---|
| Аматори | Люди, які їздять на велосипеді для відпочинку та прогулянок |
| Спортсмени | Любителі та напівпрофесіонали — шосейники, MTB-гравці |
| Батьки | Купують велосипеди та аксесуари для дітей |

**Мова інтерфейсу:** Українська (основна), можливість перемикання на англійську  
**Валюта:** Гривня (UAH)

## 2. Функціональні вимоги

### 2.1 Каталог товарів

- Категорії: велосипеди (гірські, шосейні, міські, дитячі, електро), запчастини (гальма, трансмісія, колеса, рама тощо), аксесуари (шоломи, замки, насоси, сумки)
- Фільтрація: за ціною, брендом, розміром рами, кольором, наявністю на складі
- Пошук по назві, артикулу, бренду (Elasticsearch)
- Картка товару: фото (галерея), опис, характеристики, ціна, наявність, *(відгуки — Фаза 2)*
- Позначки: «Хіт продажів», «Новинка», «Знижка»
- Сумісність запчастин (для деяких категорій)

### 2.2 Кошик та оформлення замовлення

- Додавання/видалення товарів, зміна кількості
- Збереження кошика для авторизованих та неавторизованих користувачів
- Промокоди та знижки
- Крок 1 — контактні дані (ПІБ, телефон, email)
- Крок 2 — доставка (Нова Пошта, Укрпошта, кур'єр по місту, самовивіз)
- Крок 3 — оплата
- Крок 4 — підтвердження замовлення

### 2.3 Оплата

- Карткою онлайн (WayForPay)
- Накладений платіж (Нова Пошта)
- Оплата частинами (Monobank «Покупки частинами», PrivatBank «Оплата частинами»)

### 2.4 Доставка

- Інтеграція з API Нової Пошти (пошук відділень, трекінг)
- Укрпошта
- Кур'єрська доставка по місту (власна або партнерська)
- Самовивіз з фізичних точок (якщо є)

### 2.5 Акаунт користувача

- Реєстрація / вхід (email + пароль;) Google OAuth — Фаза 2
- Особистий кабінет: замовлення, адреси доставки, список бажань, відгуки
- Трекінг статусу замовлення
- Повторне замовлення

### 2.6 Адміністративна панель

- Управління товарами (CRUD): назва, опис, фото, ціна, кількість
- Управління категоріями та фільтрами
- Управління замовленнями (статуси: нове → в обробці → відправлено → доставлено → скасовано)
- Управління користувачами та ролями (призначення `ROLE_MANAGER` / `ROLE_ADMIN` — лише для адміна)
- Управління промокодами та акціями
- Управління банером та контентом головної сторінки
- Управління поверненнями та рефандами (доступно менеджеру й адміну)

> **Розмежування доступу.** Адмін-панель доступна ролям `ROLE_MANAGER` і `ROLE_ADMIN`. Менеджер працює з операційними розділами (товари, замовлення, повернення, залишки), але не має доступу до управління користувачами/ролями, промокодами, контентом і системними налаштуваннями — вони доступні лише адміну.

### 2.7 Управління складом

#### Резервування товару

- При додаванні товару до кошика — м'яке резервування (soft hold) на 30 хвилин
- При підтвердженні замовлення — жорстке резервування (hard reserve) до відправки
- Якщо користувач не завершив checkout протягом 30 хвилин — резервування знімається, залишок повертається до загального пулу
- Фонове завдання (scheduled job) кожні 5 хвилин очищає протерміновані резерви

#### Залишки та сповіщення

- Відображення реального залишку на картці товару: «В наявності», «Залишилось мало (< 5 шт.)», «Немає в наявності»
- Кнопка «Сповістити, коли з'явиться» для товарів out-of-stock (зберігає email, надсилає сповіщення при поповненні)
- Адмін отримує email-алерт, коли залишок товару падає нижче порогового значення (налаштовується)

#### Поповнення та інвентаризація

- Адмін може вручну коригувати залишок товару з обов'язковою причиною (поповнення, інвентаризація, списання)
- Лог змін залишків зберігається для аудиту

#### Edge cases

- Якщо два користувачі одночасно додають останній товар до кошика — перший отримує резерв, другому показується повідомлення «Товар щойно закінчився»
- Якщо soft reserve протермінував (користувач тримав кошик відкритим понад 30 хвилин і не починав checkout), а товар за цей час зарезервував інший покупець — при спробі перейти до checkout користувач отримує повідомлення про недоступність товару.

### 2.8 Повернення та рефанд

#### Загальні умови

- Відповідає вимогам Закону України «Про захист прав споживачів» (ст. 9): покупець має право повернути товар належної якості протягом 14 днів з дати отримання (за умови збереження товарного вигляду, пломб, упаковки)

#### Процес повернення

1. Покупець ініціює повернення через особистий кабінет або звертається до підтримки
2. Менеджер (`ROLE_MANAGER`) підтверджує або відхиляє запит (з поясненням) протягом 2 робочих днів
3. Покупець відправляє товар зручним способом (Нова Пошта за рахунок магазину, якщо причина — брак або помилка магазину)
4. Після отримання та перевірки товару продавцем — повернення коштів.

#### Повернення коштів

- Часткові повернення (наприклад, повернення частини позицій замовлення) — підтримуються
- Статус "Повернення коштів" відображається в особистому кабінеті

#### Статуси повернення

`initiated` → `approved` / `rejected` → `received` → `refunded`

## 3. Ключові бізнес-сценарії придбання товару

Цей розділ описує найважливіші шляхи придбання товару очима покупця та бізнесу — без технічних деталей.

### 3.1 Успішне придбання товару (основний шлях)

**Сценарій.** Покупець знаходить потрібний велосипед або запчастину через каталог, фільтри чи пошук, переглядає картку товару з фото, характеристиками та реальною наявністю, додає товар у кошик і оформлює замовлення за чотири кроки: контактні дані → доставка → оплата → підтвердження. Він обирає доставку Новою Поштою, оплачує карткою онлайн і одразу отримує email-підтвердження з номером замовлення. Далі він відстежує статус замовлення в особистому кабінеті аж до отримання.

**Бізнес-цінність.** Це основний потік.

### 3.2 Покупка без реєстрації (гостьове замовлення)

**Сценарій.** Покупець, який не хоче створювати акаунт, оформлює замовлення як гість — достатньо вказати ПІБ, телефон, email та адресу доставки. Кошик зберігається навіть для неавторизованого користувача, тож він може повернутися й завершити покупку. Після оформлення йому пропонується (необов'язково) створити акаунт в один клік.

**Бізнес-цінність.** Знижує тертя на шляху до покупки і зменшує відсоток покинутих кошиків — обов'язкова реєстрація є однією з головних причин відмов на checkout.

### 3.3 Оплата при отриманні (накладений платіж)

**Сценарій.** Покупець, який не готовий платити онлайн наперед, обирає накладений платіж Новою Поштою. Замовлення підтверджується і резервується, а оплата відбувається при отриманні у відділенні.

**Бізнес-цінність.** Накладений платіж — звична й найбільш довірена форма оплати для значної частини українських покупців; його наявність прямо впливає на кількість замовлень. Водночас він несе ризик невикупу, який бізнес має відстежувати.

### 3.4 Покупка товару частинами

**Сценарій.** Покупець обирає велосипед і не готовий сплатити всю суму одразу. На кроці оплати він обирає оплату частинами (PrivatBank) і завершує покупку у звичному для нього банківському сценарії.

**Бізнес-цінність.** Розстрочка робить позиції доступнішими, підвищує середній чек (AOV) і дозволяє продавати товари верхнього цінового сегмента.

### 3.5 Невдала оплата та відновлення замовлення

**Сценарій.** Під час онлайн-оплати щось іде не так — банк відхиляє платіж, минає час очікування або платіжний шлюз тимчасово недоступний. Покупець бачить зрозуміле повідомлення (а не технічну помилку), його кошик і дані не втрачаються, а система пропонує повторити спробу або обрати інший спосіб оплати (наприклад, накладений платіж).

**Бізнес-цінність.** Зберігає продаж, який інакше був би втрачений через технічну заминку, і підтримує довіру до магазину.

### 3.6 Конкуренція за останній товар на складі

**Сценарій.** Двоє покупців майже одночасно намагаються купити останню одиницю товару. Товар закріплюється за тим, хто першим його зарезервував; другий покупець одразу бачить чесне повідомлення «Товар щойно закінчився» і може підписатися на сповіщення про надходження. Поки активне замовлення не оплачене занадто довго, резерв тримається, а потім автоматично звільняється для інших покупців.

**Бізнес-цінність.** Виключає продаж товару, якого фактично немає (overselling), що інакше призвело б до скасувань, рефандів і негативу. Точна наявність формує довіру до магазину.

### 3.7 Застосування промокоду чи участь в акції

**Сценарій.** Покупець вводить промокод у кошику або купує товар із позначкою «Знижка»/«Акція». Знижка прозоро застосовується до суми замовлення ще до оплати, і покупець бачить підсумкову економію.

**Бізнес-цінність.** Промокоди й акції — інструмент залучення, утримання та стимулювання покупок; впливають на конверсію та середній чек.

### 3.8 Очікування товару, якого немає в наявності

**Сценарій.** Покупець зацікавлений у товарі, якого зараз немає. Замість того щоб піти, він натискає «Сповістити, коли з'явиться» і залишає email. Щойно товар поповнюється на складі, система автоматично сповіщає всіх, хто очікував.

**Бізнес-цінність.** Захоплює попит і контактні дані зацікавлених покупців, повертає продажі, які інакше були б втрачені, і дає сигнал про затребувані позиції для закупівлі.

### 3.9 Повернення товару та повернення коштів

**Сценарій.** Покупець вирішує повернути товар належної якості протягом 14 днів або стикається з браком. Він ініціює повернення в особистому кабінеті, менеджер розглядає запит протягом двох робочих днів, покупець відправляє товар, і після перевірки кошти повертаються (можливе й часткове повернення окремих позицій). Статус повернення видно в кабінеті на кожному етапі.

**Бізнес-цінність.** Виконує вимоги Закону «Про захист прав споживачів», знижує юридичні ризики й формує довіру; прозорий процес повернення підвищує готовність купувати повторно.

## 5. Архітектура — Моноліт

### 5.1 Обґрунтування вибору монолітної архітектури

На початковому етапі монолітна архітектура є оптимальним вибором з таких причин:

**Швидкість розробки та простота деплою**
- Один репозиторій, одна точка деплою — менше операційних накладних витрат
- Немає необхідності налаштовувати міжсервісну комунікацію (gRPC, message broker) на старті
- Локальний запуск через `docker-compose up` без оркестрації

```
Wishlist         Delivery            ProductImage
                    │                    │
                 Payment           StockReservation
                    │
                 Return ──────── RefundTransaction
```

**Ключові таблиці:**
- `users` — покупці та адміни
- `products` — товари (назва, ціна, опис, кількість, slug)
- `product_attributes` — характеристики (розмір рами, матеріал тощо)
- `stock_reservations` — резерви товарів з TTL
- `stock_audit_log` — лог усіх змін залишків
- `orders` — замовлення зі статусами
- `order_items` — позиції замовлення
- `returns` — запити на повернення
- `refund_transactions` — транзакції рефандів
- `reviews` — відгуки (лише для підтверджених покупців)
- `promo_codes` — промокоди

### 5.5 Безпека

- **Аутентифікація:** JWT Access Token (15 хв) + Refresh Token (30 днів) у HttpOnly cookie
- **Авторизація:** Role-based (ROLE_USER, ROLE_MANAGER, ROLE_ADMIN)
    - `ROLE_USER` — покупець: власні замовлення, профіль, повернення, wishlist, відгуки
    - `ROLE_MANAGER` — співробітник: обробка замовлень, підтвердження/відхилення повернень і рефандів, коригування залишків, перегляд товарів і клієнтів (без управління користувачами та системними налаштуваннями)
    - `ROLE_ADMIN` — повний доступ: усі права менеджера + управління користувачами й ролями, промокодами, контентом головної сторінки, налаштуваннями системи
- **HTTPS:** обов'язково (Let's Encrypt)
- **CSRF:** захист для форм; для REST API — перевірка Origin
- **Rate Limiting:** Nginx + Bucket4j (Spring) для захисту від брутфорсу
- **Валідація вхідних даних:** Jakarta Validation на всіх endpoint-ах

### 5.6 Обробка помилок та edge cases в оплаті

#### Сценарії збоїв

| Сценарій | Поведінка системи |
|---|---|
| Платіж відхилено банком | Показати зрозуміле повідомлення («Оплата не пройшла. Перевірте дані картки або спробуйте інший спосіб оплати»), замовлення залишається зі статусом `pending_payment` |
| Timeout під час оплати (>30 сек) | Сесія оплати скасовується, резерв товару зберігається ще 15 хвилин, пропонується повторна спроба |
| Webhook від платіжного шлюзу не надійшов | Фоновий job перевіряє статус незавершених транзакцій через polling API шлюзу кожні 5 хвилин |
| Подвійне списання (duplicate payment) | Idempotency key на кожен запит до платіжного шлюзу; якщо подвійний платіж підтверджено — автоматичний рефанд другої транзакції |
| Часткова оплата при оплаті частинами | Якщо банк схвалив тільки частину суми — замовлення не підтверджується, користувач отримує сповіщення |
| Платіж пройшов, але замовлення не створилось | Транзакційна обгортка: спочатку фіксується payment, потім order. При збої — rollback з автоматичним рефандом |
| Шлюз недоступний (503) | Retry з exponential backoff (3 спроби); якщо недоступний — показати повідомлення і запропонувати накладений платіж |

#### Технічна реалізація

- Усі платіжні операції — в окремих транзакціях БД з ізоляцією `SERIALIZABLE`
- Idempotency key — стабільний на одну логічну спробу оплати: генерується клієнтом при ініціації платежу і не змінюється при ретраях (наприклад, `paymentAttemptId` (UUID), збережений у БД на час сесії оплати). Включати `timestamp` не можна — інакше кожен ретрай дає новий ключ і захист від дублів не спрацьовує.
- Статуси транзакцій: `initiated` → `processing` → `success` / `failed` / `refunded`
- Усі callback/webhook від платіжних шлюзів перевіряються підписом (HMAC)
- Логування кожного кроку платіжного флоу для аудиту

### 5.7 Моніторинг, логування та резервне копіювання

#### Логування

- Структуроване логування (JSON) усіх запитів та помилок; кореляційний `traceId` наскрізь через запит
- Окремий аудит-лог для чутливих операцій: зміни залишків (`stock_audit_log`), платіжні транзакції, зміни ролей користувачів, обробка повернень
- Рівні логів конфігуруються; PII у логи не пишеться (телефон/email маскуються)

#### Моніторинг та алерти

- **Збір показників стану системи.** Бекенд сам звітує про себе (скільки запитів обробив, як швидко, скільки впало з помилкою) — (Spring Boot Actuator + Micrometer). Prometheus збирає ці показники й зберігає їхню історію.
- **Візуалізація.** На основі зібраних даних будуються графіки й панелі у Grafana — для технічних метрик з розділу 10.3: аптайм, швидкість відповіді (p95), відсоток помилок (5xx), частка успішних webhook від платіжних шлюзів.
- **Автоматичні сповіщення (алерти).** Щоб не стежити за графіками вручну. Пороги: відсоток помилок > 0.5%, швидкість відповіді (p95) > 300 мс, недоступність платіжного шлюзу, підтвердження оплат перестали доходити.
- **Зовнішня перевірка доступності.** Застосунок має спеціальну технічну адресу (health-check), яка на запит відповідає, чи він працює.

#### Резервне копіювання та відновлення

- Автоматичний бекап PostgreSQL: щодоби повна копія + безперервне збереження змін між ними (WAL-archiving)
- Зберігання бекапів в окремому S3-бакеті з версіонуванням; ретенція — не менше 30 днів
- Дані замовлень зберігаються 5 років (вимога бухобліку, розділ 7.2): окрема політика довготривалого архівування.

## 6. SEO-вимоги

### 6.1 Технічне SEO

- **Meta-теги:** унікальні `<title>` та `<meta description>` для кожної сторінки (шаблони для категорій і карток товарів)
- **Canonical URLs:** для сторінок з фільтрами та пагінацією (`?page=2`, `?color=red`) — canonical вказує на базовий URL категорії
- **Hreflang:** якщо підтримується англійська версія — `hreflang="uk"` та `hreflang="en"`
- **robots.txt:** закривається `/admin`, `/checkout`, `/cart`, `/account`; дозволяється `/catalog`, `/product`
- **sitemap.xml:** автоматично генерується при зміні каталогу (продукти, категорії); включає `lastmod`, `priority`, `changefreq`; подається в Google Search Console
- **Швидкість:** відповідає вимогам Core Web Vitals — LCP ≤ 2.5 сек, CLS < 0.1, INP < 200 мс

### 6.2 URL-структура

- Людиночитабельні slug-и: `/catalog/girski-velosypedy`, `/product/trek-marlin-7-2024`
- Slug генерується транслітерацією з назви при створенні товару; зміна slug — редирект 301 зі старого URL
- Пагінація через параметр `?page=N` (не через окремі URL)
- Фільтри — через query params (`?brand=trek&color=chornyi`), не через path segments

### 6.3 Контентне SEO

- H1 — одна на сторінку: назва категорії або товару
- Alt-теги для всіх зображень товарів (заповнюються адміном при завантаженні)
- Унікальний опис для кожного товару (не дублювати з виробником)
- Open Graph теги (`og:title`, `og:description`, `og:image`) для шерингу в соцмережах

### 6.4 Структуровані дані (Schema.org / JSON-LD)

Розмітка у форматі JSON-LD вбудовується на стороні сервера (SSG/SSR) для коректного зчитування пошуковими ботами та формування rich snippets у видачі.

- **`Product`** — на картці товару: `name`, `image`, `description`, `sku`, `brand`, `category`
- **`Offer`** (вкладений у `Product`) — `price`, `priceCurrency` (UAH), `availability` (`InStock` / `OutOfStock` / `LimitedAvailability` відповідно до реального залишку), `url`
- **`AggregateRating`** + **`Review`** — рейтинг і відгуки товару (підключається разом з модулем відгуків, Фаза 2)
- **`BreadcrumbList`** — хлібні крихти на сторінках категорій і товарів (Головна → Категорія → Товар)
- **`Organization`** — на головній: назва, логотип, контакти
- **`WebSite`** + **`SearchAction`** — для sitelinks search box у Google

> Розмітку слід валідувати через Google Rich Results Test перед релізом. `availability` має синхронізуватися з реальним статусом залишку (розділ 2.7), щоб у видачі не показувалася неактуальна наявність.

## 7. Захист персональних даних (GDPR / Закон України)

### 7.1 Правова база

Обробка персональних даних регулюється:
- **Законом України «Про захист персональних даних»** (ЗУ № 2297-VI)

### 7.2 Принципи обробки

- Збираємо тільки ті дані, які необхідні для виконання замовлення (ПІБ, телефон, email, адреса доставки)
- Персональні дані не передаються третім сторонам, окрім: служб доставки (Нова Пошта), платіжних шлюзів, сервісів email-розсилки — і тільки в обсязі, необхідному для виконання послуги
- Термін зберігання: дані акаунту — до видалення користувачем; дані замовлень — 5 років (вимоги бухгалтерського обліку)

### 7.3 Технічна реалізація

- **Cookie consent banner** при першому відвіданні: «Необхідні» (завжди) / «Аналітика» / «Маркетинг» (за вибором)
- Налаштування cookie-уподобань доступні в будь-який момент через футер
- Google Analytics 4 підключається тільки після згоди на аналітику
- Email-розсилки маркетингового характеру — тільки за явною згодою (opt-in при реєстрації або окремим checkbox)
- **Unsubscribe-посилання** в кожному маркетинговому листі

### 7.4 Права суб'єктів даних

Реалізується через особистий кабінет та/або звернення до підтримки:

| Право | Реалізація |
|---|---|
| Доступ до своїх даних | Сторінка «Мої дані» в кабінеті |
| Виправлення даних | Редагування профілю |
| Видалення акаунту | Кнопка «Видалити акаунт» → анонімізація даних (збереження замовлень без прив'язки до особи) |

### 7.5 Документи

- Публічна **Політика конфіденційності** (посилання в футері)
- **Публічна оферта** / Умови використання
- Форма згоди при реєстрації з посиланням на Політику конфіденційності
